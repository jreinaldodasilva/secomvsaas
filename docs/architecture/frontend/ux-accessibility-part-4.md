# Secom Frontend — UX, Accessibility & Visual Design Audit
## Part 4: High-Level Recommendations & Improvement Priorities

> Final part of the Secom UX audit. See Parts 1–3 for all findings.
> This section consolidates all recommendations from the audit into a prioritized, actionable reference.

---

## 10. High-Level Recommendations & Improvement Priorities

### 10.1 Priority Classification

Recommendations are classified by the same severity scale used throughout this document:

- 🟥 **Critical** — Blocks usage or violates accessibility standards. Address before next public release.
- 🟧 **High** — Strong usability, consistency, or accessibility issue. Address in next sprint.
- 🟨 **Medium** — UX friction, polish, or modernity gap. Address in backlog.
- 🟩 **Low** — Optimization or enhancement opportunity. Address when convenient.

---

### 10.2 🟥 Critical Recommendations

#### C1 — Fix Modal Duplicate `id="modal-title"`

**Finding:** `Modal.tsx` uses static string IDs `"modal-title"` and `"modal-desc"` for `aria-labelledby` and `aria-describedby`. When `CrudPage` renders both the form modal and the delete `ConfirmDialog` simultaneously, duplicate IDs exist in the DOM, breaking ARIA associations for screen readers.

**Recommendation:** Replace static IDs with values generated by `useId()`. Pass the generated ID to the `aria-labelledby` attribute and to the `id` of the title element.

**Effort:** Low. **Impact:** High — affects all 7 domain module workflows.

---

#### C2 — Inject `aria-describedby` in `FormField`

**Finding:** `FormField` renders an error paragraph with `id="{name}-error"` but does not inject `aria-describedby` onto its child input. Domain forms use raw `<input>` elements that do not set this attribute manually. Screen readers do not programmatically associate field errors with their inputs.

**Recommendation:** `FormField` should inject `aria-describedby` onto its child element when an error is present. Use `React.cloneElement` to add the attribute, or adopt a context-based approach where the child reads the error ID from context.

**Effort:** Medium. **Impact:** High — affects all 7 domain forms and all auth forms using `FormField`.

---

#### C3 — Replace Hybrid Validation Messages with Portuguese-Only Messages

**Finding:** Zod's built-in English error messages are appended to i18n field name keys, producing strings like `"Título — String must contain at least 5 character(s)"` visible to end users in a Portuguese-language government application.

**Recommendation:** Override Zod's default error map with a Portuguese error map at the application level. All validation messages should be in Portuguese and use non-technical language appropriate for government service users.

**Effort:** Medium. **Impact:** High — affects all 7 domain module forms and all users who encounter validation errors.

---

#### C4 — Restore Focus on Modal Close

**Finding:** `Modal` moves focus to the first focusable element on open but does not restore focus to the trigger element on close. Keyboard and screen reader users lose their position in the page after closing a modal.

**Recommendation:** Store a `ref` to `document.activeElement` before the modal opens. In the cleanup effect (when `isOpen` transitions from `true` to `false`), call `.focus()` on the stored element.

**Effort:** Low. **Impact:** High — affects all modal interactions across the application.

---

### 10.3 🟧 High Recommendations

#### H1 — Fix Citizen Portal Nav Touch Targets

**Finding:** `.navLink` in `CitizenPortalLayout.module.css` has no `min-height`. Citizen portal navigation links are approximately 20px tall — well below the WCAG 2.5.5 minimum of 44px. This is a direct barrier for mobile users, who are the primary audience for citizen-facing services.

**Recommendation:** Add `min-height: var(--touch-target-min)` and `display: flex; align-items: center` to `.navLink` in `CitizenPortalLayout.module.css`.

**Effort:** Low. **Impact:** High — affects all mobile citizen users.

---

#### H2 — Fix Pagination Button Touch Targets

**Finding:** `DataTable` pagination buttons have `height: 2.25rem` (36px) — below the 44px WCAG 2.5.5 minimum.

**Recommendation:** Increase pagination button height to `var(--button-height-md)` (44px) or add `min-height: var(--touch-target-min)`.

**Effort:** Low. **Impact:** Medium — affects mobile users navigating paginated lists.

---

#### H3 — Add `no_show` and `failed` to `StatusBadge`

**Finding:** `AppointmentsPage` uses `no_show` and `SocialMediaPage` uses `failed` as status values. Neither is in `StatusBadge`'s `STATUS_LABELS` map — users see raw enum strings in the table.

**Recommendation:** Add Portuguese labels and token-mapped colors for `no_show` ("Não compareceu") and `failed` ("Falhou") to `StatusBadge`'s `STATUS_LABELS` and `STATUS_VARIANT` maps.

**Effort:** Low. **Impact:** High — visible to all users of Appointments and Social Media modules.

---

#### H4 — Resolve Toast `aria-live` Conflict

**Finding:** `ToastContainer` has `aria-live="polite"` while individual `Toast` components have `role="alert"` (which implies `aria-live="assertive"`). This conflicting live region semantics produces unpredictable screen reader announcement behavior.

**Recommendation:** Remove `aria-live` from `ToastContainer`. Keep `role="alert"` on individual `Toast` components. This ensures each toast is announced assertively without the container creating a second live region.

**Effort:** Low. **Impact:** Medium — affects all screen reader users receiving toast notifications.

---

#### H5 — Replace Emoji Icons in Citizen Portal with SVG Icons

**Finding:** The citizen portal uses emoji (`🏛️`, `📅`, `📋`, `📢`, `🏗️`, `👤`, `👋`, `ℹ️`) as primary icons throughout `CitizenPortalHomePage`, `CitizenDashboardPage`, and `CitizenPortalLayout`. The staff dashboard uses the `Icon` component (react-icons SVG set) consistently. This creates a visual quality gap and reduces institutional credibility.

**Recommendation:** Replace emoji with `Icon` component instances using appropriate icon names from the existing react-icons set. The `Icon` component already supports `aria-hidden` and `aria-label` props.

**Effort:** Low. **Impact:** Medium — affects brand perception and visual consistency.

---

#### H6 — Add Layout Wrapper to Error Pages

**Finding:** `/unauthorized` and `*` (404) render without any layout wrapper — no header, no footer, no navigation context. The 404 "back to home" button hard-codes `/admin/dashboard`, which redirects citizen users to the staff login page.

**Recommendation:** Wrap error pages in `PublicLayout` to provide navigation context. Update the 404 "back to home" link to `/` (landing page) — a neutral destination appropriate for all user types.

**Effort:** Low. **Impact:** Medium — affects all users who encounter error pages.

---

#### H7 — Add `role="alert"` to `LoginForm` Error Display

**Finding:** The `LoginForm` error `<div>` has no `role="alert"` or `aria-live` attribute. Screen readers will not announce login errors when they appear.

**Recommendation:** Add `role="alert"` to the error banner `<div>` in `LoginPage.tsx`. The `errorBanner` class in `Auth.module.css` is already used — only the ARIA attribute is missing.

**Effort:** Low. **Impact:** Medium — affects all staff users who encounter login errors.

---

#### H8 — Remove Dead Production Dependencies

**Finding:** `framer-motion` and `react-hot-toast` are listed as production dependencies but are not imported anywhere in the source tree. `framer-motion` adds ~100KB gzipped to the production bundle.

**Recommendation:** Remove both packages from `package.json` `dependencies`. Remove the `motion` chunk from `vite.config.ts` `manualChunks`.

**Effort:** Low. **Impact:** Medium — reduces bundle size and eliminates dependency confusion.

---

### 10.4 🟨 Medium Recommendations

#### M1 — Add `inputMode` and `type` Attributes to Domain Form Fields

**Finding:** Phone fields, CPF fields, and URL fields in domain forms use `type="text"` with no `inputMode` hint. On mobile, this presents a standard QWERTY keyboard instead of the appropriate numeric or URL keyboard.

**Recommendation:**
- Phone fields: `type="tel"` or `inputMode="tel"`
- CPF fields: `inputMode="numeric"` with `pattern="[0-9]*"`
- URL fields: `type="url"`

**Effort:** Low. **Impact:** Medium — improves mobile data entry for `atendente` and `assessor` roles.

---

#### M2 — Add `autoComplete` Attributes to Citizen-Facing Forms

**Finding:** Citizen registration and profile forms do not set `autoComplete` attributes. Browser autofill is disabled, requiring citizens to manually type all fields on mobile.

**Recommendation:** Add `autoComplete` hints to `CitizenRegisterPage` fields: `name`, `email`, `new-password`. For the citizen profile display, `autoComplete` is not applicable (read-only).

**Effort:** Low. **Impact:** Medium — improves citizen registration experience on mobile.

---

#### M3 — Add Specific Approval Workflow Feedback Messages

**Finding:** When a press release status changes (e.g., `draft` → `review`), the user receives a generic "Salvo com sucesso" toast. For a government approval workflow, specific feedback would improve clarity.

**Recommendation:** In `PressReleasesPage`, detect status changes in `buildPayload` and pass a status-specific success message to `savedMessage` (e.g., "Comunicado enviado para revisão" when status changes to `review`).

**Effort:** Medium. **Impact:** Medium — improves workflow clarity for `assessor` and `admin` roles.

---

#### M4 — Add Role Indicator to Sidebar Footer

**Finding:** The sidebar footer shows the user's name but not their role. Users unfamiliar with the system may not understand why certain modules are unavailable.

**Recommendation:** Add a small role badge or label below the user's name in the sidebar footer (e.g., "Assessor", "Atendente"). Use the existing `StatusBadge` component or a simple styled `<span>`.

**Effort:** Low. **Impact:** Medium — improves role awareness for new staff users.

---

#### M5 — Tokenize Breakpoints

**Finding:** Breakpoints are hardcoded values in media queries across multiple CSS files. Changing a breakpoint requires a grep-and-replace across the codebase.

**Recommendation:** Define breakpoint values as CSS custom properties (e.g., `--bp-mobile: 480px`, `--bp-tablet: 768px`, `--bp-desktop: 1024px`) in `tokens/index.css`. Note: CSS custom properties cannot be used directly in `@media` queries — this would require a build-time approach (PostCSS custom media queries) or documentation-only standardization.

**Effort:** Medium. **Impact:** Low-Medium — improves maintainability for future responsive design changes.

---

#### M6 — Add Skeleton Loading to Citizen Portal Pages

**Finding:** `CitizenDashboardPage` and `CitizenProfilePage` show no skeleton loading state during data fetch. The citizen name renders as empty/undefined briefly during context initialization.

**Recommendation:** Add `Skeleton` component instances to citizen portal pages for the welcome banner name and profile field values. Use the existing `Skeleton` component with `variant="text"`.

**Effort:** Low. **Impact:** Low-Medium — improves perceived performance for citizen users.

---

#### M7 — Standardize Loading Prop Naming

**Finding:** Loading state is expressed with three different prop names: `isLoading` (`Button`, `ConfirmDialog`), `loading` (`Input`), and `isPending` (`FormComponentProps`).

**Recommendation:** Standardize on `isLoading` across all components. Update `Input` and `FormComponentProps` to use `isLoading`.

**Effort:** Low. **Impact:** Low — improves contributor experience and API consistency.

---

#### M8 — Add `aria-label="Carregando"` to Loading Component

**Finding:** `Loading.tsx` uses `aria-label="Loading"` in English in a Portuguese-language application.

**Recommendation:** Change to `aria-label="Carregando"` or use the i18n `t('common.loading')` key.

**Effort:** Low. **Impact:** Low — minor accessibility improvement for screen reader users.

---

#### M9 — Add `:active` State to Interactive Elements

**Finding:** No component defines an explicit `:active` state beyond the browser default. On touch devices, `:active` provides important tactile feedback.

**Recommendation:** Add `transform: scale(0.97)` on `:active` for `Button` and interactive `Card` components. Respect `prefers-reduced-motion` by omitting the transform when motion is reduced.

**Effort:** Low. **Impact:** Low-Medium — improves touch feedback on mobile.

---

### 10.5 🟩 Low Recommendations

#### L1 — Add `ConnectionBanner` ARIA Live Region

**Finding:** `ConnectionBanner` has no `role="alert"` or `aria-live` attribute. Screen reader users are not notified when the API becomes unreachable.

**Recommendation:** Add `role="alert"` to the `ConnectionBanner` container element.

**Effort:** Low. **Impact:** Low — affects screen reader users during API outages.

---

#### L2 — Expand Citizen Portal with Appointments View

**Finding:** The citizen portal currently offers only 2 authenticated pages (dashboard and profile). Citizens cannot view their appointments through the portal.

**Recommendation:** Add a `/portal/appointments` read-only page showing the citizen's scheduled appointments. This requires a backend endpoint returning appointments filtered by citizen ID.

**Effort:** High. **Impact:** High — significantly improves citizen portal utility.

---

#### L3 — Add Self-Service Profile Editing to Citizen Portal

**Finding:** `CitizenProfilePage` is read-only. Citizens cannot update their own data.

**Recommendation:** Add an edit mode to `CitizenProfilePage` with a form for updating name, phone, and address fields. Requires a backend `PATCH /api/v1/citizen-auth/profile` endpoint.

**Effort:** High. **Impact:** High — core citizen portal functionality.

---

#### L4 — Add Dark Mode Support

**Finding:** No dark mode is implemented. The token system uses semantic surface tokens (`--color-bg-primary`, `--color-bg-secondary`) that would support a dark mode via a `:root[data-theme="dark"]` override.

**Recommendation:** Define a dark mode token override set in `tokens/index.css` under `[data-theme="dark"]`. Add a theme toggle to the user profile settings. Respect `prefers-color-scheme: dark` as the default.

**Effort:** High. **Impact:** Medium — improves user experience for users who prefer dark mode.

---

#### L5 — Add Bottom Navigation for Citizen Portal Mobile

**Finding:** The citizen portal header navigation is not optimized for one-handed mobile use. All navigation items are in the top header, requiring a reach to the top of the screen.

**Recommendation:** Consider a bottom navigation bar for the citizen portal on mobile (≤767px) with icons and labels for the main sections (Home, Profile, Appointments). This is a common pattern for citizen-facing government mobile services.

**Effort:** Medium. **Impact:** Medium — improves mobile ergonomics for citizen users.

---

#### L6 — Add `prefers-contrast: high` Support

**Finding:** The `Card` component already has a `prefers-contrast: high` rule (`border-width: 2px`). This pattern should be extended to other interactive elements.

**Recommendation:** Add `@media (prefers-contrast: high)` rules to `Button`, `Input`, `Modal`, and `DataTable` to increase border widths and ensure sufficient contrast in high-contrast mode.

**Effort:** Medium. **Impact:** Low — improves accessibility for users with visual impairments who use high-contrast mode.
