## Quick Win Details

### QW-01: Fix Modal Duplicate `id="modal-title"`

**Problem:** `Modal.tsx` uses static string IDs `"modal-title"` and `"modal-desc"` for `aria-labelledby` and `aria-describedby`. When `CrudPage` renders both the form modal and the delete `ConfirmDialog` simultaneously, two elements with `id="modal-title"` exist in the DOM. The HTML spec requires unique IDs — screen readers and ARIA tooling associate `aria-labelledby` with the first matching ID, producing incorrect modal title announcements.

**Impact:** All screen reader users interacting with any of the 7 domain module forms or confirm dialogs receive incorrect ARIA associations. This is a WCAG 4.1.1 (Parsing) violation affecting every modal interaction in the application.

**Effort:** 0.5 days

**Suggested Approach:** Replace the static `"modal-title"` and `"modal-desc"` strings with values generated by React's `useId()` hook inside `Modal.tsx`. Pass the generated IDs to the `aria-labelledby` and `aria-describedby` attributes and to the `id` props of the title and description elements.

**Expected Outcome:** Unique ARIA IDs on every modal instance. Screen readers correctly announce the title of whichever modal is in focus. WCAG 4.1.1 violation resolved.

**Source Part:** Part 2 §5.2 (Issue A1), Part 4 §10.2 (C1)

### QW-02: Restore Focus on Modal Close

**Problem:** `Modal.tsx` moves focus to the first focusable element on open (via focus trap) but does not store a reference to the trigger element and restore focus on close. After closing any modal, keyboard focus falls to `<body>` — users must re-navigate from the top of the page.

**Impact:** Every keyboard and screen reader user loses their navigation position after closing any modal across all 7 domain modules, all confirm dialogs, and the session timeout modal. This is a WCAG 2.4.3 (Focus Order) violation.

**Effort:** 0.5 days

**Suggested Approach:** In `Modal.tsx`, capture `document.activeElement` into a `ref` before the modal opens. In the cleanup effect triggered when `isOpen` transitions from `true` to `false`, call `.focus()` on the stored element reference.

**Expected Outcome:** Keyboard focus returns to the element that triggered the modal (e.g., the "Editar" button in the DataTable row) after every modal close. WCAG 2.4.3 violation resolved.

**Source Part:** Part 2 §5.2 (Issue A5), Part 4 §10.2 (C4)

### QW-03: Add `role="alert"` to `LoginForm` Error Display

**Problem:** The `LoginForm` error `<div>` (styled with `errorBanner` class in `Auth.module.css`) has no `role="alert"` or `aria-live` attribute. When a login attempt fails, the error message appears visually but is not announced by screen readers.

**Impact:** Screen reader users who enter incorrect credentials receive no feedback that their login failed. They must navigate the page to discover the error — a direct usability barrier at the application entry point for all staff roles.

**Effort:** 2 hours

**Suggested Approach:** Add `role="alert"` to the error banner `<div>` in `LoginPage.tsx`. The `errorBanner` CSS class is already applied — only the ARIA attribute is missing.

**Expected Outcome:** Screen readers announce login errors immediately when they appear. WCAG 4.1.3 (Status Messages) violation resolved for the login flow.

**Source Part:** Part 2 §5.4, Part 4 §10.3 (H7)

### QW-04: Resolve Toast `aria-live` Conflict

**Problem:** `ToastContainer` has `aria-live="polite"` while individual `Toast` components have `role="alert"` (which implies `aria-live="assertive"`). These conflicting live region semantics produce unpredictable screen reader announcement behavior — some screen readers may announce toasts twice, others may suppress them.

**Impact:** All screen reader users receiving toast notifications (success, error, and info messages after any CRUD operation) experience unreliable announcements. This affects every staff role after every save, delete, or error action.

**Effort:** 2 hours

**Suggested Approach:** Remove the `aria-live="polite"` attribute from `ToastContainer`. Retain `role="alert"` on individual `Toast` components. Each toast will then be announced assertively as a self-contained live region without the container creating a conflicting second region.

**Expected Outcome:** Consistent, predictable screen reader announcements for all toast notifications. No double-announcement or suppression.

**Source Part:** Part 2 §5.4, Part 4 §10.3 (H4)

### QW-05: Fix Citizen Portal Nav Touch Targets to ≥44px

**Problem:** `.navLink` in `CitizenPortalLayout.module.css` has no `min-height` defined. Citizen portal navigation links are approximately 20px tall — well below the WCAG 2.5.5 minimum of 44px. This is the primary navigation for the public-facing portal, used predominantly on mobile devices.

**Impact:** All mobile citizen users struggle to tap navigation links accurately. This is a direct WCAG 2.5.5 (Target Size) violation on the most citizen-facing surface of the application, where public accessibility obligations are highest.

**Effort:** 2 hours

**Suggested Approach:** Add `min-height: var(--touch-target-min)` and `display: flex; align-items: center` to the `.navLink` rule in `CitizenPortalLayout.module.css`. The `--touch-target-min` token is already defined as 44px.

**Expected Outcome:** Citizen portal navigation links meet WCAG 2.5.5 on all mobile viewports. WCAG 2.5.5 violation resolved for the citizen portal.

**Source Part:** Part 3 §8.1, Part 4 §10.3 (H1)

### QW-06: Fix Pagination Button Touch Targets to ≥44px

**Problem:** `DataTable` pagination buttons have `height: 2.25rem` (36px) — 8px below the WCAG 2.5.5 minimum of 44px. Pagination is used on every domain module page with more than one page of results.

**Impact:** Mobile users on any paginated list (press releases, media contacts, clippings, events, appointments, citizen records, social media posts) cannot reliably tap pagination controls. Affects all staff roles and atendente on mobile devices.

**Effort:** 2 hours

**Suggested Approach:** Increase pagination button height to `var(--button-height-md)` (44px) or add `min-height: var(--touch-target-min)` to the pagination button rule in `DataTable.module.css`.

**Expected Outcome:** Pagination buttons meet WCAG 2.5.5 on all mobile viewports across all 7 domain module pages.

**Source Part:** Part 3 §8.1, Part 4 §10.3 (H2)

### QW-07: Add `no_show` and `failed` to `StatusBadge`

**Problem:** `AppointmentsPage` uses `no_show` and `SocialMediaPage` uses `failed` as status values. Neither is present in `StatusBadge`'s `STATUS_LABELS` map. Users see the raw enum strings `"no_show"` and `"failed"` as literal text in the table — an unfinished appearance visible to all users of these modules.

**Impact:** Atendente users viewing appointment records and social media managers viewing post statuses see non-human-readable strings. Screen reader users hear `"no_show"` and `"failed"` as meaningless identifiers. Affects the two modules most likely to surface these statuses in daily workflows.

**Effort:** 2 hours

**Suggested Approach:** Add Portuguese labels (`"Não compareceu"` for `no_show`, `"Falhou"` for `failed`) and token-mapped color variants to `StatusBadge`'s `STATUS_LABELS` and `STATUS_VARIANT` maps. No structural changes to the component are required.

**Expected Outcome:** All status values across all 7 modules display human-readable Portuguese labels. Raw enum strings are eliminated from the UI.

**Source Part:** Part 1 §2.2, Part 3 §9.2, Part 4 §10.3 (H3)

### QW-08: Replace Emoji Icons in Citizen Portal with SVG `Icon` Component

**Problem:** `CitizenPortalHomePage`, `CitizenDashboardPage`, and `CitizenPortalLayout` use emoji (`🏛️`, `📅`, `📋`, `📢`, `🏗️`, `👤`, `👋`, `ℹ️`) as primary icons. The staff dashboard uses the `Icon` component (react-icons SVG set) consistently. This creates a visible quality gap between the two surfaces and reduces the institutional credibility of the public-facing portal.

**Impact:** Citizens and government stakeholders perceive the citizen portal as lower quality than the staff dashboard. Emoji rendering varies across operating systems and browsers, creating inconsistent visual output. The institutional tone expected of a government communication secretariat is undermined.

**Effort:** 1 day

**Suggested Approach:** Identify the appropriate icon names from the existing react-icons set already used in the staff dashboard. Replace each emoji instance with an `Icon` component instance using `aria-hidden={true}` (the adjacent text label provides the accessible name). No new dependencies are required — the `Icon` component and react-icons are already in use.

**Expected Outcome:** Visual consistency between citizen portal and staff dashboard. Institutional credibility of the citizen-facing surface is restored. Icon rendering is consistent across all browsers and operating systems.

**Source Part:** Part 1 §2.2, Part 2 §5.6, Part 4 §10.3 (H5)

### QW-09: Add Layout Wrapper to Error Pages; Fix 404 Destination

**Problem:** The `/unauthorized` and `*` (404) pages render without any layout wrapper — no header, footer, or navigation context. The 404 "back to home" button hard-codes `/admin/dashboard`, which redirects citizen users to the staff login page rather than the landing page.

**Impact:** Users who encounter error pages (wrong URL, expired session, permission denied) have no visual continuity with the rest of the application and no clear path back. Citizens sent to 404 are redirected to the staff login — a confusing and broken experience.

**Effort:** 1 day

**Suggested Approach:** Wrap both error pages in `PublicLayout` to provide header and footer context. Update the 404 "back to home" link target from `/admin/dashboard` to `/` (the landing page) — a neutral destination appropriate for all user types regardless of role.

**Expected Outcome:** Error pages are visually consistent with the rest of the application. All users have a clear, role-appropriate path back from error states.

**Source Part:** Part 1 §2.2, Part 2 §4.2, Part 4 §10.3 (H6)

### QW-10: Remove Dead Production Dependencies

**Problem:** `framer-motion` and `react-hot-toast` are listed in `package.json` `dependencies` but are not imported anywhere in the source tree. `framer-motion` is included in the `motion` manual chunk in `vite.config.ts`, adding approximately 100KB gzipped to the production bundle on every initial page load.

**Impact:** All users pay a ~100KB bundle tax for a library that provides no functionality. The `motion` chunk is loaded as part of the initial bundle split, increasing time-to-interactive. Contributors may attempt to use these packages assuming they are intentionally included, creating future confusion.

**Effort:** 2 hours

**Suggested Approach:** Remove `framer-motion` and `react-hot-toast` from `package.json` `dependencies`. Remove the `motion` entry from the `manualChunks` configuration in `vite.config.ts`. Run the build to confirm no import errors.

**Expected Outcome:** ~100KB reduction in production bundle size. Cleaner dependency tree. No contributor confusion about available animation libraries.

**Source Part:** Part 3 §6.2, Part 3 §8.5, Part 4 §10.3 (H8)

### QW-11: Add Role Indicator to Sidebar Footer

**Problem:** The sidebar footer displays the logged-in user's name but not their role. Staff users unfamiliar with the RBAC system cannot understand why certain modules or actions are unavailable to them — there is no persistent UI signal indicating their current role.

**Impact:** New staff members (assessor, atendente, social_media) may be confused when they cannot access modules visible to admin users. Support requests and onboarding friction increase without a visible role indicator.

**Effort:** 2 hours

**Suggested Approach:** Add a small role label or badge below the user's name in the sidebar footer. Use the existing `StatusBadge` component or a simple styled `<span>` with a neutral variant. Map role keys to human-readable Portuguese labels (e.g., `assessor` → "Assessor", `atendente` → "Atendente").

**Expected Outcome:** All staff users can see their current role at a glance in the sidebar. Onboarding friction and support requests related to missing features are reduced.

**Source Part:** Part 3 §9.1, Part 4 §10.4 (M4)

### QW-12: Add Skeleton Loading to Citizen Portal Pages

**Problem:** `CitizenDashboardPage` and `CitizenProfilePage` show no skeleton loading state during data fetch. The citizen name renders as empty or undefined briefly during context initialization, creating a visible layout flash on every page load.

**Impact:** Citizens experience a jarring flash of empty content on the two pages they use most. This is particularly noticeable on slower mobile connections — the primary access method for citizen-facing government services.

**Effort:** 1 day

**Suggested Approach:** Add `Skeleton` component instances (already available in the component library with `variant="text"`) to the welcome banner name placeholder and profile field values in both citizen portal pages. Conditionally render skeletons while the auth context or data query is loading.

**Expected Outcome:** Citizen portal pages display a smooth skeleton loading state instead of empty/undefined content flashes. Perceived performance improves on all connection speeds.

**Source Part:** Part 3 §7.1, Part 4 §10.4 (M6)

### QW-13: Add `inputMode` and `type` Attributes to Domain Form Fields

**Problem:** Phone fields, CPF fields, and URL fields in all 7 domain module forms use `type="text"` with no `inputMode` hint. On iOS and Android, this presents a standard QWERTY keyboard instead of the appropriate numeric or URL keyboard, requiring extra taps to switch keyboard modes.

**Impact:** Atendente users entering citizen phone numbers and CPF values, and assessor users entering media contact phone numbers and clipping URLs, must manually switch keyboard modes on mobile. This is a daily friction point for the two roles most likely to use domain forms on mobile devices.

**Effort:** 1 day

**Suggested Approach:** Apply targeted attribute additions to the relevant raw `<input>` elements in domain forms: `type="tel"` or `inputMode="tel"` for phone fields; `inputMode="numeric"` with `pattern="[0-9]*"` for CPF fields; `type="url"` for URL fields. No component changes are required — these are HTML attribute additions to existing inputs.

**Expected Outcome:** Mobile users see the appropriate keyboard type for each field. Data entry speed and accuracy improve for phone, CPF, and URL fields across all domain modules.

**Source Part:** Part 3 §8.3, Part 4 §10.4 (M1)

### QW-14: Add `autoComplete` Attributes to Citizen Registration Form

**Problem:** The citizen registration form (`CitizenRegisterPage`) does not set `autoComplete` attributes on any fields. Browser autofill is effectively disabled, requiring citizens to manually type their name, email, and password on mobile — the primary device for citizen-facing government services.

**Impact:** Citizens registering on mobile must type all fields manually. Browser password managers cannot offer to save credentials. This increases registration abandonment risk and friction at the primary citizen onboarding entry point.

**Effort:** 2 hours

**Suggested Approach:** Add `autoComplete` hints to `CitizenRegisterPage` fields: `autoComplete="name"` for the name field, `autoComplete="email"` for the email field, `autoComplete="new-password"` for the password field. These are standard HTML attribute additions requiring no component changes.

**Expected Outcome:** Browser autofill works on the citizen registration form. Password managers can offer to save credentials. Registration friction on mobile is reduced.

**Source Part:** Part 3 §8.3, Part 4 §10.4 (M2)

### QW-15: Fix `aria-label="Loading"` to Portuguese

**Problem:** The `Loading` component uses `aria-label="Loading"` in English. In a Portuguese-language government application, all ARIA labels should be in Portuguese to provide a consistent screen reader experience.

**Impact:** Screen reader users hear "Loading" in English when the full-page loading spinner is displayed (during route lazy-load and auth initialization). Minor but observable inconsistency in a government application where language consistency is a credibility signal.

**Effort:** 1 hour

**Suggested Approach:** Change `aria-label="Loading"` to `aria-label="Carregando"` in `Loading.tsx`, or replace the hardcoded string with the existing i18n key `t('common.loading')` if the translation key is already defined.

**Expected Outcome:** All ARIA labels in the application are in Portuguese. Screen reader users receive a consistent language experience throughout.

**Source Part:** Part 2 §5.4, Part 4 §10.4 (M8)

### QW-16: Standardize Loading Prop Naming to `isLoading`

**Problem:** Loading state is expressed with three different prop names across the component library: `isLoading` (`Button`, `ConfirmDialog`), `loading` (`Input`), and `isPending` (`FormComponentProps`). Contributors must check each component's API to know the correct prop name.

**Impact:** Inconsistent component API increases contributor cognitive load and creates a risk of incorrect prop usage (e.g., passing `isLoading` to `Input` which expects `loading`). This is a design system debt item that compounds as the codebase grows.

**Effort:** 1 day

**Suggested Approach:** Standardize on `isLoading` as the canonical loading prop name across all components. Update `Input` to accept `isLoading` (with a deprecation alias for `loading` if backward compatibility is needed) and update `FormComponentProps` to use `isLoading`. Update all call sites.

**Expected Outcome:** Single, predictable loading prop name across the entire component library. Contributor onboarding is simplified. Risk of incorrect prop usage is eliminated.

**Source Part:** Part 4 §10.4 (M7)

### QW-17: Add `:active` State to `Button` and Interactive `Card`

**Problem:** No component defines an explicit `:active` state beyond the browser default. On touch devices, the `:active` pseudo-class provides the primary tactile feedback signal — without it, interactive elements feel unresponsive to touch.

**Impact:** All mobile users (staff and citizens) receive no visual confirmation that a button or card tap has registered. This is particularly noticeable on slower devices where the action response may be delayed. Affects every interactive element in the application.

**Effort:** 1 day

**Suggested Approach:** Add `transform: scale(0.97)` on `:active` for `Button` (in `global.css` `.btn:active`) and interactive `Card` components. Wrap the transform in a `@media (prefers-reduced-motion: no-preference)` guard to respect user motion preferences. The existing `prefers-reduced-motion` global rule already handles the override.

**Expected Outcome:** Interactive elements provide immediate visual feedback on touch. Mobile UX feels more responsive and polished across all user roles.

**Source Part:** Part 3 §6.3, Part 4 §10.4 (M9)

### QW-18: Add `ConnectionBanner` ARIA Live Region

**Problem:** `ConnectionBanner` has no `role="alert"` or `aria-live` attribute. When the API becomes unreachable, the banner appears visually at the top of the page but screen reader users receive no notification of the connectivity loss.

**Impact:** Screen reader users continue attempting actions (form submissions, data loads) without knowing the API is unavailable. They discover the problem only when actions fail silently or produce confusing error states.

**Effort:** 1 hour

**Suggested Approach:** Add `role="alert"` to the root element of `ConnectionBanner`. This is a single attribute addition — no structural changes are required.

**Expected Outcome:** Screen reader users are immediately notified when API connectivity is lost. They can stop attempting actions and wait for the connection to be restored.

**Source Part:** Part 2 §5.4, Part 4 §10.5 (L1)
